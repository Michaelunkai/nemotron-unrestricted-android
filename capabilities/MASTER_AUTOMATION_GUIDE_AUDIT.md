## Guide prescriptions corrected for Android, safety, or truthful verification

- Termux uses Android app lifecycle and JobScheduler facilities, not `systemd-resolved` or a general-purpose systemd service manager.
- Docker-based `act`, Docker builds, Bazel, raw block-device operations, kernel/LSM bypasses, and guaranteed raw-packet features are capability-gated and are not advertised on an unrooted phone.
- Android runtime permissions, Accessibility, battery exemptions, Shizuku authorization, authentication, file/share pickers, payments, device lock, and secure surfaces remain user/platform gates.
- FTP is not an automatic fallback because it lacks the identity and confidentiality required for PC automation. Verified gateway, strict SSH/SFTP, pinned HTTPS, or signing-required SMB are used instead.
- `shred` and `wipe` cannot guarantee physical erasure on flash media. Encryption, limited retention, recoverable retirement, and key destruction are the valid controls.
- Daily package upgrades, global language-package updates, and routine SSH-key rotation can break a working environment or invalidate trust. The app reports/stages changes, binds them to an exact plan hash, preserves relevant state, refuses removals, and verifies health.
- Plugin discovery never executes arbitrary code. Custom hooks are explicit task steps with schemas, consent classification, postconditions, and rollback metadata.
- Clipboard, gallery, contacts, SMS, telephony, microphone, camera, location, and UI automation never run as ambient surveillance; each action follows its typed permission/privacy/consent contract.

## Completion rule

A guide item is complete only when its route exists, every eligible model can discover it, permission/consent/failure behavior is truthful, focused tests pass, and an independent readback proves the requested effect. Documentation, an installed binary, an exit code, a generic progress sentence, or a command row alone is not completion evidence.
